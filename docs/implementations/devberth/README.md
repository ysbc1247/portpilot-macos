# DevBerth implementation state

DevBerth is a native, local-only macOS developer utility. Runtime process data flows through injected discovery, process-control, Docker, launch, persistence, security, and health-check services. Verified metadata is retained separately from inferred labels and restart suggestions.

Implemented vertical slices:

- TCP/UDP IPv4/IPv6 discovery with tagged `lsof`, process enrichment, classification, project inference, rolling metadata cache, live diffs, history, notifications, and Docker correlation.
- Dense customizable Active Ports table, process inspector, guarded graceful/force actions, reviewed profile import, command palette, native sidebar, menu bar, accessibility labels, light/dark appearance, and app icon.
- SwiftData schema/migration plan, Keychain secrets, direct/login-shell launching, expected-port conflict/readiness, HTTP health checks, bounded redacted persistent logs, auto-launch, and dependency-layer project orchestration.
- Docker unavailable/daemon/running states, JSON container parsing, published ports, Compose labels, stop/restart, and recent logs.
- Bounded, cycle-safe process lineage; deterministic confidence-labeled ownership conclusions; live managed-runtime reconciliation; bounded ownership-evidence persistence; and a “Why is this running?” inspector.
- Owner-aware lifecycle dispatch for managed services, Docker containers, guarded host processes, Kubernetes port forwards, and SSH tunnels. Exact Compose scope uses service-wide commands; an unavailable Compose scope falls back only to exact-container Stop/Restart. Homebrew paths, supervisor ancestry, and parent PID 1 remain inferred classification while strong same-user listener owners use guarded instance control rather than an unverified manager command.
- Independent service controls across Projects, Managed Services, Runtime, and Sessions, with visible in-flight/results state. Exact expected-port observations can be stopped only after explicit confirmation and fresh owner/fingerprint/listener revalidation; multiple ports owned by one exact routed target are stopped once, graceful timeout revalidates before force escalation, and the observation never grants start or restart authority. Stop All attempts every service in reverse dependency order and aggregates exact failures. Destructive confirmations share an Escape-dismissible sheet with a native cancel shortcut.
- Four explicit restart-trust states, exact configuration digests, V4 validation persistence, ordinary-launch gating, isolated start/readiness/controlled-stop validation, guided observed-process conversion, and reliable restart of registered verified managed runtimes.
- Keychain-only secret-like environment fields with staged rollback, independent references on duplication, and reference-aware cleanup on edit or deletion.
- Automated parser/domain/persistence/health/security tests plus harmless real-process discovery and termination integration tests.

Validated locally on 2026-07-22 with the repository's warnings-as-errors macOS test action: 187 of 187 tests passed with zero failures, skips, or expected failures in 109 seconds. The run covered nine harmless real-process integrations, 19 MCP control-plane tests, all seven UI tests, exact-container Compose fallback, case-only path acceptance, true symlink refusal, guarded inferred-owner stops, revalidated force escalation, Stop All continuation after a failed target, Escape dismissal of destructive confirmations, schema migrations, trust drift, Keychain lifecycle, process identity/group safety, deterministic ownership, and exact managed restart. Live QA through the rebuilt installed MCP helper returned the production project/runtime snapshots, classified the actual Cloud gateway as `guardedExternalProcess` with “Stop this observed process instance,” and verified both `sanggwon-cloud-backend` Compose contexts, including the case-varying override path. No live service was mutated during this read-only QA.

The Phase 2 product identity is DevBerth. `ProductIdentity` and `ProductDataMigrator` preserve the legacy store, log, defaults, and Keychain compatibility boundary. The private GitHub repository retains its legacy name until a separate rename is authorized.

Phase 2 domain vocabulary now separates transient `ObservedListener`/`ObservedProcess` evidence from durable `ManagedServiceConfiguration`. V1 `LaunchProfileRecord` naming remains only at the persistence and current-UI compatibility boundary.

The additive V2 schema has independent records for runtime instances, ownership evidence, restart trust, lifecycle events, discovery metadata, sessions, session services, and restore results. V3 adds full fingerprints and controlled-group evidence; V4 adds exact managed-service validation results. Genuine previous-schema fixtures preserve existing data without inventing verification. See `Documentation/DOMAIN_MODEL.md` and `Documentation/RESTART_TRUST_MODEL.md`.

The ownership and restart-reliability slices are complete at their current boundaries. Exact managed registrations and Docker container metadata choose their owner-layer controllers; external instance actions retain the full fingerprint/listener revalidation path. Existing profiles require an exact successful validation before ordinary launch, and inferred processes use the guided conversion workflow for restart. Compose manager commands, Homebrew actions, and launchd actions remain unavailable pending exact controller context, while exact-container and guarded same-user instance fallbacks remain deliberately narrower. Runtime lifecycle, health, and incident intelligence are implemented as described above.
